§1 — IP Addressing, NAT & Routing

Q: Do routers also have IP addresses? And if they do, are they also a subject of packets?

Yes. Routers have multiple network interfaces (ports), and each interface gets its own IP address — one per connected network.

Example: A home router has a LAN-side IP (e.g. 192.168.1.1) facing local devices, and a WAN-side IP assigned by the ISP.

Are routers subjects of packets?

  • Passing-through traffic — the router is NOT the destination. It reads the destination IP, decrements the TTL, and forwards onward.
  • Direct traffic (e.g. pinging your gateway, traceroute) — the router IS the destination.
  • Routing protocol traffic (OSPF, BGP) — routers exchange packets with each other directly as sender and receiver.
Key terms: Network Interface · Default Gateway · Routing Table · TTL · ICMP

Q: Does a domain like google.com really map to a single IP / single machine?

No. Several mechanisms distribute traffic across many machines:

  • DNS returning multiple IPs: A DNS query for google.com can return a different IP each time, rotating across servers.
  • Anycast: The same IP is announced from many geographic locations; routing sends you to the nearest one.
  • Load balancers: One public IP fronts many backend machines. The load balancer distributes incoming connections.
  • GeoDNS: DNS returns different IPs depending on where in the world you're querying from.

Google combines all three — GeoDNS, Anycast, and internal load balancing — to handle global scale.

Key terms: DNS · Anycast · Load Balancer · GeoDNS

Q: Is it true every machine has a unique IP address?

Only partially. The answer depends on which layer you're looking at:

  • Public IPs are globally unique — no two devices on the public internet share one.
  • Private IPs (e.g. 192.168.x.x, 10.x.x.x) are intentionally reused across millions of private networks. They are only unique within their local network.
  • NAT (Network Address Translation) is what makes this work — your router translates between your private IP and the single public IP your ISP gives you.
Key terms: Public IP · Private IP · NAT · RFC 1918

Q: How are IP addresses assigned? At manufacture?

IPs are not baked into hardware. They are assigned dynamically and administratively through a hierarchy:

  • IANA (Internet Assigned Numbers Authority) manages the global pool.
  • RIRs (Regional Internet Registries — e.g. ARIN, RIPE) allocate blocks to ISPs by region.
  • ISPs assign IPs to customers, often dynamically via DHCP.
  • Your router assigns private IPs to your devices via its own DHCP server.

What IS fixed at manufacture is the MAC address — a hardware identifier that operates at Layer 2, not Layer 3. It has nothing to do with internet routing.

Key terms: IANA · RIR · DHCP · MAC Address · Layer 2 vs Layer 3

Q: So my laptop doesn't have an IP address?

It actually has two simultaneously:

  • A private IP assigned by your router (e.g. 192.168.1.5) — used within your local network.
  • It indirectly shares the router's public IP via NAT — that's what the rest of the internet sees.

From the internet's perspective, your laptop is hidden behind the router. The router keeps a NAT table to track which internal device sent which request, so returning packets reach the right machine.

Key terms: Private IP · Public IP · NAT Table · Router

Q: Is the router's public IP what's seen externally? Could the router itself be behind another router?

Yes — this happens with Carrier-Grade NAT (CGNAT). Some ISPs run their own NAT layer, giving home routers private IPs. The ISP's router holds the real public IP, meaning your home router is itself behind a NAT.

  • This means multiple households can share a single public IP.
  • Port forwarding (for hosting a server at home) breaks under CGNAT because you don't control the outer router.
  • CGNAT is increasingly common due to IPv4 address exhaustion, and is a key reason IPv6 adoption matters.
Key terms: CGNAT · IPv4 Exhaustion · Port Forwarding · Double NAT

Q: Is IPv6 the same as IPv4 but with more characters?

The address space is larger (128-bit vs 32-bit), but IPv6 also introduces meaningful architectural changes:

  • No NAT required — every device can have a globally unique public IP.
  • No router fragmentation — oversized packets are handled via Path MTU Discovery; routers drop and notify rather than fragment.
  • Built-in IPSec — security was designed in, not bolted on.
  • Multicast replaces broadcast — more efficient for local network communication.
  • SLAAC (Stateless Address Autoconfiguration) — devices can generate their own IP without a DHCP server.
  • Dual-stack transition — IPv4 and IPv6 are incompatible, so the internet runs both simultaneously during the long migration.
Key terms: IPv6 · 128-bit · SLAAC · IPSec · Dual-Stack · Multicast

Q: Must a server have its own public IP with no NAT bottlenecks?

Not strictly — a server only needs to be reachable. Options:

  • Port forwarding through NAT can work for small/home servers — you forward a specific port on the router to the server's private IP.
  • CGNAT breaks this — you can't port-forward through an ISP-controlled outer NAT you don't have access to.
  • Professional servers typically have their own public IPs (or sit behind load balancers that do) for reliability, control, and no dependency on NAT cooperation.
Key terms: Port Forwarding · CGNAT · Load Balancer · Public IP

§2 — Packets & Data Transmission

Q: When a packet is sent, what happens if the data exceeds the maximum packet size?

Every network link has a Maximum Transmission Unit (MTU) — the largest packet size it can carry. Standard Ethernet MTU is 1500 bytes.

IPv4 — Fragmentation:

  • Routers along the path break the oversized packet into smaller fragments.
  • Each fragment carries the same destination IP, plus a fragment offset and flags (e.g. "More Fragments") so they can be reassembled.
  • The Don't Fragment (DF) flag can be set by the sender — if set and the packet is too large, the router drops it and sends back an ICMP "Fragmentation Needed" error instead of fragmenting.
  • Reassembly happens at the destination host, not at intermediate routers.
  • If any single fragment is lost, the entire original packet must be retransmitted.

IPv6 — Path MTU Discovery (PMTUD):

  • IPv6 routers do NOT fragment. If a packet is too large, the router drops it and sends back an ICMPv6 "Packet Too Big" message to the sender.
  • The sender reduces its packet size and retries. This is called Path MTU Discovery.
Key difference: IPv4 lets routers fragment (burden on the network); IPv6 forces the sender to handle it (burden on the endpoint), keeping routers simpler and faster.
Key terms: MTU · Fragmentation · Fragment Offset · DF Flag · Path MTU Discovery · ICMPv6 Packet Too Big

Q: Why does TCP use byte positions for sequencing rather than packet numbers?

Because TCP has no fixed packet boundaries. Packet-based numbering would break under real conditions:

  • Retransmissions can be different sizes — a lost segment may be retransmitted as a different number of packets than the original.
  • MTU changes mid-connection — segmentation can change due to path changes or congestion control.
  • Nagle's algorithm — small writes can be coalesced into one segment; the segment boundaries are not fixed.

Byte numbering reflects TCP's core abstraction: it is a pure byte stream, not a packet stream. The receiver doesn't care how the bytes were split — only that they arrive in order and complete.

  • Sequence number wraparound: Numbers are 32-bit and wrap around (modular arithmetic handles this correctly).
  • PAWS (Protection Against Wrapped Sequences) — a timestamp extension that prevents old duplicate segments from being confused with new ones after wraparound.
Key terms: TCP · Byte Stream · Sequence Number · Nagle's Algorithm · PAWS · Retransmission

§3 — Distributed Systems & Servers

Q: How do distributed services handle databases and cookies across multiple machines? How do code updates work without downtime?

Databases:

  • All machines share a database cluster — not separate per-machine copies.
  • Replication: A primary node handles writes; read replicas serve read queries for scale.
  • Sharding: The dataset is split across multiple nodes so no single machine holds all the data.

Sessions & Cookies:

  • Sticky sessions: The load balancer always routes a user to the same server, so session state stays local.
  • Centralized session store: Sessions stored in a shared cache (e.g. Redis) accessible by all servers.
  • Stateless tokens (JWT): The session state is encoded in the token itself — no server-side storage needed.

Zero-downtime deployments:

  • Rolling update: Update one machine at a time; traffic continues on the rest.
  • Blue-Green deployment: Run two identical environments; switch all traffic from old (blue) to new (green) instantly.
  • Canary deployment: Gradually shift a small percentage of traffic to the new version; expand if stable, roll back if not.
Also relevant: CAP Theorem — a distributed system can guarantee at most two of: Consistency, Availability, Partition Tolerance.
Key terms: Replication · Sharding · Redis · JWT · Rolling Update · Blue-Green · Canary · CAP Theorem

§4 — Protocols & the Internet's Nature

Q: Is the internet more of a protocol/format globally agreed upon, rather than a physical thing? Can protocols be updated globally? Does the kernel manage protocols so programmers don't have to?

Nature of the internet: Yes — the internet is fundamentally a set of agreements (protocols) about how data should be formatted and routed, implemented across independently owned physical infrastructure.

How protocols are updated:

  • Governed by IETF (internet protocols), W3C (web standards), IEEE (hardware/link layer) via published RFCs.
  • Updates are voluntary and gradual — no one can force adoption. Backward compatibility is required.
  • Layers are independent — an update at one layer doesn't break others (in principle).
  • Protocol ossification is a real problem: TCP has barely changed since the 1970s because middleboxes (firewalls, NAT devices) that inspect TCP headers resist change. QUIC was built on UDP specifically to bypass this — by running inside UDP, it avoids being blocked or mangled by TCP-aware middleboxes.

Kernel managing protocols:

  • Yes — the full TCP/IP network stack lives in the kernel. Applications use the socket API and never touch raw TCP/IP.
  • A protocol change requires kernel/OS updates, sometimes hardware driver updates, sometimes application updates.
  • User-space networking (e.g. DPDK, QUIC implemented in browsers) is an emerging exception — it bypasses the kernel stack entirely for performance or flexibility.
Key terms: IETF · RFC · W3C · IEEE · Protocol Ossification · QUIC · Socket API · User-space Networking

§5 — P2P Networks

Q: Is Telegram a peer-to-peer network?

No. Telegram is centralized — all messages route through Telegram's servers, even with end-to-end encryption enabled (in Secret Chats).

  • Standard chats are encrypted in transit but stored on Telegram's servers.
  • Secret Chats use end-to-end encryption but the packets still physically pass through Telegram's infrastructure.
  • True P2P apps (e.g. Briar, Session) route messages directly between users or through a volunteer node network, with no central server.
Key terms: Centralized Architecture · End-to-End Encryption · P2P · Briar · Session

Q: Do P2P chat apps exist? How do they handle floating IP addresses? How did Napster handle it?

Napster's approach: Used a central index server that tracked which users had which files. The index was centralized; only the file transfers were P2P. This was its legal and architectural weakness.

Modern solution — DHT (Distributed Hash Table):

  • The index itself is distributed across all nodes — no central server.
  • Each node is responsible for tracking a subset of the data, identified by hash.
  • Used by BitTorrent, and underlies decentralized chat/storage systems.

Real P2P apps:

  • Briar — routes through Tor; works even without internet (Bluetooth/WiFi direct).
  • Session — uses a volunteer-run node network (no central server).
  • Matrix — federated (many independently run servers, not strictly P2P but decentralized).
Key terms: DHT · Napster · Bootstrap Node · BitTorrent · Briar · Session · Matrix · Federation

Q: If all BitTorrent nodes go offline and reconnect with new IPs, wouldn't the hash tables be worthless? Doesn't there need to be a fixed address somewhere?

Yes — and BitTorrent acknowledges this. No system is truly free of fixed reference points:

  • Bootstrap nodes: BitTorrent clients are hardcoded with the addresses of a few known bootstrap nodes. These introduce you to the DHT network when you first connect.
  • Bootstrap nodes only provide initial peer introductions — they don't hold content or control the network. The actual DHT lives across all connected nodes.
  • Peer caching: Clients store recently seen peer IPs on disk. On reconnect, they try cached peers first before falling back to bootstrap nodes, further reducing reliance on fixed addresses.

The key insight: BitTorrent minimizes the power of fixed reference points (they can't censor or control content), but doesn't eliminate them entirely.

Key terms: DHT · Bootstrap Node · Peer Caching · Decentralization

§6 — Sockets & Ports

Q: A socket is an abstraction of a pipeline between two machines, unique per IP pair and port. How does a kernel differentiate packets arriving at the same port from many clients?

By the 5-tuple — the full identifier of a connection:

  • Protocol (TCP/UDP)
  • Source IP
  • Source Port
  • Destination IP
  • Destination Port

Two clients connecting to port 443 on the same server will have different source IPs and/or source ports, making their 5-tuples unique. The kernel maps each incoming packet to the correct socket by matching its 5-tuple.

  • C10K problem: Handling 10,000+ simultaneous connections efficiently. Solved by event-driven I/O models (epoll on Linux, kqueue on BSD/macOS) rather than a thread-per-connection model.
  • At Google's scale, the real solution is distributing traffic across many machines before any single port is overwhelmed.
Key terms: 5-Tuple · Socket · epoll · kqueue · C10K Problem

Q: What is the actual point of ports if the 5-tuple does the real differentiation work?

Ports and the 5-tuple solve different problems at different levels:

  • Ports route packets to the right process. When a packet arrives, the kernel uses the destination port to hand it to the correct listening application (e.g. port 80 → web server, port 22 → SSH daemon).
  • The 5-tuple identifies the right connection within that process. A single web server process on port 80 may manage thousands of simultaneous connections — the 5-tuple is how it tells them apart.
  • Well-known ports (80, 443, 22, etc.) are a global convention so clients know where to connect without being told in advance.
Key terms: Well-Known Ports · Ephemeral Ports · Process · 5-Tuple · Socket

Q: What happens if two separate browser processes connect to google.com simultaneously from the same machine?

Each process is assigned a different ephemeral source port by the OS. Even though both connect to the same destination IP and port (e.g. 142.250.x.x:443), their 5-tuples remain unique because the source port differs.

  • Returning packets are matched to the right process by destination port (which equals the ephemeral source port the process was assigned).
  • Within a process, multiple connections to the same server are distinguished by file descriptor — each accepted connection gets its own fd.
Key terms: Ephemeral Port · File Descriptor · 5-Tuple · OS Port Assignment

Q: So a socket is made between processes, and ports make that possible?

Close — with a small refinement: a socket connects a local process to a remote endpoint (not two local processes). The full routing hierarchy, from most general to most specific:

  • IP address — which machine?
  • Port — which process on that machine?
  • 5-tuple — which connection within that process?
  • File descriptor — which socket handle inside the process's code?

Each level answers a progressively more specific question about where data belongs.

Key terms: Socket · Port · 5-Tuple · File Descriptor · Endpoint
